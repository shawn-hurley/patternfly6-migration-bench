# TC012: Chip â†’ Label Migration â€” Missing `variant="outline"` Analysis

## The Problem

When migrating `Chip` to `Label`, the tool correctly renames the component (`Chipâ†’Label`, `ChipGroupâ†’LabelGroup`) but does not add `variant="outline"`. The pf-codemods baseline adds `variant="outline"` to each Label to preserve the visual appearance of the original Chip.

## The Migration

| PF5 | Tool Output | pf-codemods Output |
|---|---|---|
| `<Chip>Apple</Chip>` | `<Label>Apple</Label>` | `<Label variant="outline">Apple</Label>` |

## Why `variant` Is Not Required

The `variant` prop on Label is **optional** with a default of `"filled"`:

```typescript
variant?: 'outline' | 'filled' | 'overflow' | 'add'  // default: 'filled'
```

The tool's output compiles and runs correctly. `<Label>Apple</Label>` renders a valid Label component â€” it just renders with the **filled** appearance (solid colored background) instead of the **outline** appearance (transparent background, gray border) that matches the old Chip's visual treatment.

## What the Rule Already Provides

The generated rule message includes explicit appearance guidance:

```
IMPORTANT: Chip did not have a variant prop. Label defaults to a different 
appearance. Consider which variant value matches Chip's look: add, outline.
```

This guidance is generated by the `infer_appearance_notes()` function in the rule generator, which compares CSS modifier data between the old component (Chip) and the new component (Label):

1. For each variant value on Label, it checks whether the corresponding CSS modifier class (`pf-m-{value}`) exists on the old Chip component
2. If the modifier exists on Label but NOT on Chip, it is a **candidate** for matching Chip's default appearance
3. If only 1 candidate remains, generates a directive: "Add variant='X' to preserve visual parity"
4. If 2+ candidates remain, generates an advisory: "Consider which variant value matches Chip's look"

For Chip to Label, the analysis finds **2 candidates** (`outline` and `add`), so it generates the advisory rather than a directive. The algorithm cannot narrow further because both `pf-m-outline` and `pf-m-add` have structurally similar CSS (both override BackgroundColor, BorderColor, BorderWidth, Color).

## Why the Algorithm Finds 2 Candidates

| Label variant | CSS modifier | On old Chip? | On new Label? | Classification |
|---|---|---|---|---|
| `filled` | `pf-m-filled` | No | **No** (it's the default -- no modifier needed) | Skip |
| `outline` | `pf-m-outline` | No | Yes | **Candidate** |
| `add` | `pf-m-add` | No | Yes | **Candidate** |
| `overflow` | `pf-m-overflow` | Yes | Yes | Shared (not a default match) |

Both `outline` and `add` are new modifier classes that Chip didn't have. The algorithm correctly identifies them as candidates for matching Chip's default appearance, but cannot distinguish between them using CSS structure alone -- both modifiers override the same set of CSS properties (BackgroundColor, BorderColor, BorderWidth, Color) with similar structural patterns.

## Why the LLM Doesn't Act on the Guidance

The LLM (Goose) receives the appearance notes as "IMPORTANT" annotations at the end of the family migration context. However, its response focuses entirely on the rename mechanics (`Chip->Label`, `ChipGroup->LabelGroup`) and never addresses the appearance/variant guidance. This is a pattern seen across multiple test cases -- advisory notes are less reliably followed than direct fix instructions.

## What Would Be Needed to Fix This Automatically

To narrow from 2 candidates to 1, the algorithm would need to compare the old Chip's **base CSS appearance** (not modifier CSS) against each candidate modifier's resolved CSS values:

- Old Chip base: transparent background, thin gray border -- matches `pf-m-outline` (transparent bg, gray border)
- `pf-m-add`: also transparent background with border, but semantically for "add new item" UI pattern (typically shows a `+` icon)

However, the pipeline currently only extracts **modifier-specific** CSS properties (`CssModifierEffect`), not base component CSS. Extracting base component CSS (the non-modifier `.pf-v5-c-chip { ... }` declarations) would require a new extraction capability in `css_profile/mod.rs`.

## Assessment

This is a **visual fidelity** issue, not a correctness issue:

- **Compiles**: Yes -- `variant` is optional, default is valid
- **Runs at runtime**: Yes -- Label renders correctly with filled appearance
- **Visual difference**: Yes -- filled blue tag instead of outlined gray tag
- **Functional difference**: No -- same content, same click/close behavior

In a real migration, the visual change from Chip (outlined) to Label (filled) would be noticed during visual/QA testing and fixed manually. The automated tool correctly performs the component rename, which is the primary migration requirement.

## Conclusion

The missing `variant="outline"` is scored as 2 (correct but non-idiomatic) because:

1. The `variant` prop is optional -- the output is valid PF6 code
2. The rule generator already provides appearance guidance, but with 2 candidates instead of 1
3. The LLM doesn't act on the advisory guidance (a common pattern)
4. Narrowing to 1 candidate would require base component CSS extraction (not currently available)
5. The visual difference is detectable during QA testing and trivially fixable

This is accepted as a limitation of the automated migration. The tool prioritizes correctness (valid PF6 code that compiles and runs) over visual fidelity (pixel-perfect appearance preservation).
