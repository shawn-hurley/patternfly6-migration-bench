# Semver Rules Coverage Analysis — tackle2-ui Migration

## Objective

Analyze how well our semver-generated rules and fix-strategies cover the real-world PF5→PF6 migration changes observed in [konveyor/tackle2-ui PR #3246](https://github.com/konveyor/tackle2-ui/pull/3246). Produce a structured coverage report that maps each real-world change to the corresponding rule(s), identifies gaps, and recommends improvements.

This prompt is designed to be re-run with updated rule files as the semver-analyzer and fix-engine evolve.

---

These are YAML rule files generated by [semver-analyzer](https://github.com/shawn-hurley/semver-analyzer) by diffing PF5 and PF6 package APIs. They will be consumed by Kantra with the frontend-analyzer-provider to detect migration issues in user codebases.

**Location**: `<RULES_DIR>/semver_rules/`

**Files**:
| File | Contents | Rule Count (current) |
|------|----------|---------------------|

| `ruleset.yaml` | Ruleset metadata (name, description, labels) | N/A |

**Rule Structure** (YAML):
```yaml
- ruleID: <unique-identifier>
  labels:
  - source=semver-analyzer
  - change-type=<removed|renamed|css-variable|composition|dependency-update|manifest>
  - kind=<prop|interface|enum|type|...>
  - has-codemod=<true|false>
  - package=<@patternfly/react-core|...>
  - family=<ComponentName>
  effort: <1-5>
  category: mandatory
  description: <human-readable description>
  message: <detailed migration guidance>
  when:
    frontend.referenced:      # For API/component rules
      pattern: ^ComponentName$
      location: <JSX_COMPONENT|IMPORT|PROP>
      from: '@patternfly/react-core'
    # OR
    frontend.cssvar:           # For CSS variable rules
      pattern: --pf-v5-global--...
    # OR
    frontend.dependency:       # For dependency rules
      name: '@patternfly/react-core'
      upperbound: 5.99.99
```

**Key rule categories to check**:
- `change-type=removed` + `kind=prop`: Prop removals (e.g., Card.isFlat, Nav.theme)
- `change-type=renamed` + `kind=prop`: Prop renames (e.g., labelIcon→labelHelp)
- `change-type=removed` + `kind=interface`: Type/interface removals
- `change-type=composition`: Component family restructuring (e.g., EmptyStateIcon removed)
- `change-type=css-variable`: CSS variable renames
- `change-type=dependency-update`: Package upgrades
- SD-pipeline rules (ruleID prefix `sd-`): DOM structure, ARIA/role/OUIA attribute changes, composition tree changes, rendered element changes

## Analysis Procedure

### Step 1: Load and Index Inputs

1. Read `<RULES_DIR>/semver_rules/breaking-changes-api.yaml` — extract all ruleIDs, their `when` conditions, change-types, and descriptions
2. Read `<RULES_DIR>/semver_rules/breaking-changes-composition.yaml` — same
3. Read `<RULES_DIR>/semver_rules/breaking-changes-css.yaml` — same (note: ~3154 rules, focus on ones relevant to tackle2-ui changes)
4. Read `<RULES_DIR>/semver_rules/breaking-changes-deps.yaml` — same
5. Read `<RULES_DIR>/fix-guidance/fix-strategies.json` — index by ruleID
6. Read `results/tackle2-ui/tackle2-ui-changes.json` — the ground truth

### Step 2: For Each tackle2-ui Change, Find Matching Rules

For each of the 88 entries in `tackle2-ui-changes.json`, search the rule files for rules that would detect this change:

**Matching heuristics by change category:**

| Change Category | How to Find Matching Rule |
|----------------|--------------------------|
| `component-rename` | Search for ruleIDs containing the old component name (e.g., "Text", "Chip"). Check `breaking-changes-api.yaml` for `renamed` change-type and `breaking-changes-composition.yaml` for family changes. |
| `import-path-change` | Search `breaking-changes-api.yaml` for rules with `ImportPathChange` strategy in fix-strategies.json, or `breaking-changes-deps.yaml` for dependency rules. |
| `prop-rename` | Search `breaking-changes-api.yaml` for rules with `change-type=renamed` and the specific prop name in the ruleID or description. |
| `prop-removal` | Search `breaking-changes-api.yaml` for rules with `change-type=removed` and `kind=prop`. |
| `prop-value-change` | Search fix-strategies.json for `PropValueChange` entries mentioning the component and prop. |
| `structure-change` | Search `breaking-changes-composition.yaml` for the component family. Check fix-strategies.json for `CompositionChange`, `FamilyMigration`, or `ChildToProp` strategies. |
| `interface-rename` | Search `breaking-changes-api.yaml` for `kind=interface` or `kind=type` rules. |
| `token-rename` | Search `breaking-changes-api.yaml` for rules mentioning the token name (e.g., `global_palette_*`). |
| `css-class-prefix` | Search `breaking-changes-css.yaml` for `semver-css-class-rename-*` rules. |
| `css-global-token` | Search `breaking-changes-css.yaml` for `semver-consumer-css-rename-var-*` rules. |
| `css-component-variable` | Search `breaking-changes-css.yaml` for rules matching the component CSS variable. |
| `behavior-change` | Search `breaking-changes-composition.yaml` for SD-pipeline rules (prefix `sd-`) covering DOM structure, portal, context, or prop default changes. Only classify as `inherent-gap` if the change depends on runtime state not visible in the JSX AST. |
| `markup-change` | Search `breaking-changes-composition.yaml` for SD-pipeline rules covering rendered elements, ARIA/role/OUIA data-attribute changes, and wrapper element changes. The SD pipeline detects many markup changes deterministically. |
| `package-upgrade` | Search `breaking-changes-deps.yaml` for `semver-dep-update-*` rules. |

### Step 3: For Each Match, Evaluate Fix Strategy Quality

When a matching rule is found, also check if there's a corresponding fix-strategies.json entry:

- **Has rule + has correct fix-strategy**: Full coverage
- **Has rule + fix-strategy is wrong/incomplete**: Rule exists but fix guidance is incorrect
- **Has rule + no fix-strategy**: Detection only, no automated fix
- **No rule**: Gap — the change is not detectable

For fix-strategies that exist, evaluate:
- Does the `strategy` type correctly describe the change? (e.g., a prop rename should use `Rename`, not `RemoveProp`)
- Are the `from`/`to` values correct?
- Does the strategy vocabulary support this change type? (e.g., can `PropValueChange` express `alignRight`→`alignEnd`?)

### Step 4: Classify Each Change

Assign each tackle2-ui change one of these coverage statuses:

| Status | Meaning |
|--------|---------|
| `fully-covered` | Matching rule exists AND fix-strategy is correct |
| `rule-only` | Matching rule exists but no fix-strategy or fix-strategy is wrong |
| `wrong-guidance` | Rule exists but fix-strategy gives incorrect guidance |
| `missing-rule` | No rule covers this change |
| `inherent-gap` | Change depends on runtime state/timing not visible in JSX AST and cannot be expressed as a static analysis rule |

### 5. Wrong Guidance

## Important Context

### What semver-analyzer does
The [semver-analyzer](https://github.com/shawn-hurley/semver-analyzer) generates migration rules by comparing PF5 and PF6 packages across three pipelines: structural API diffing of `.d.ts` declarations (TD), deterministic source-level analysis of `.tsx` files for DOM, accessibility, composition, and CSS changes (SD), and optional LLM-based behavioral analysis (BU). It detects:
- Removed/renamed exports, props, interfaces, types, enums
- CSS variable renames (by comparing CSS custom property lists)
- CSS class renames (by comparing generated CSS class names)
- Dependency/peer-dependency changes (by comparing package.json)
- DOM structure changes (root elements, wrapper elements, rendered HTML tags)
- ARIA, role, and OUIA data-attribute changes
- Component composition tree changes (parent-child relationships, edge strength, conformance rules)
- Prop default value changes
- Portal, context, forwardRef, and memo usage changes

### What it cannot detect
- Prop value semantic changes where the type signature didn't change
- Dynamic behavioral changes that depend on runtime state, computed props, or timing not visible in the JSX AST
- Implicit composition contracts enforced only through documentation, not code

### What fix-engine-cli does
The [fix-engine-cli](https://github.com/shawn-hurley/fix-engine) takes Kantra analysis output (rule violations) and fix-strategies.json, then uses an LLM to apply the fixes. The LLM sees:
- The rule's `message` field (from the YAML rule)
- The fix-strategy entry (from fix-strategies.json)
- The source code context where the violation was found

The fix quality depends on:
1. The rule detecting the right thing (rule accuracy)
2. The fix-strategy providing correct guidance (strategy accuracy)
3. The LLM interpreting and applying the fix correctly (tool behavior)

This analysis focuses on (1) and (2). Tool behavior issues are out of scope.
